What is this

Built for private, local AI

MIRA Mini is a desktop application built on Electron. It runs Google Gemma 3 4B INT4 quantized entirely on your local hardware using node-llama-cpp — no internet required after setup.

Zero Cloud Dependency

All inference happens on-device. No data leaves your machine. Ever.

GPU Accelerated

Leverages Metal on Mac and CUDA/Vulkan on Windows via llama.cpp backend for fast token generation.

Full Chat History

All conversations are saved locally as JSON. Load, continue, or delete any dialogue from the sidebar.

Configurable Persona

System prompt, temperature, top-p — all adjustable from the built-in settings dashboard.

Deep Link Onboarding

First-run uses a browser-based onboarding flow with a custom mira:// protocol to handoff back to the app.

Live Log Chronicle

All engine events are logged to disk and streamed live to the in-app Chronicle view for debugging.

Architecture

How it works

MIRA Mini follows a strict Electron multi-process architecture. The main process owns all AI inference; the renderer is a pure UI layer with no direct Node access.

Main Process

main.js — controls the BrowserWindow, manages app lifecycle, single-instance lock, deep link handling, and IPC routing. All filesystem ops live here.

Inference Engine

inference.js — wraps node-llama-cpp. Initializes llama, loads the GGUF model, manages the context sequence and chat session, streams tokens via onTextChunk, and disposes native memory on shutdown.

Renderer (UI)

app.js — pure DOM manipulation. No Node access. Communicates exclusively through the window.mira contextBridge API exposed by the preload script.

Preload Bridge

preload.cjs — the secure bridge. Exposes a whitelist of IPC invoke/on calls to the renderer as window.mira. contextIsolation is always on.

Key Engineering Decisions

Why we built it this way and the trade-offs made:

  • Single Instance Lock — app.requestSingleInstanceLock() prevents multiple copies of the process running simultaneously (which caused the original 700MB RAM issue).
  • Engine deferred until onboarded — The heavy 4B model only initializes after the user completes the browser onboarding flow. Cold launch is instant.
  • asar: false — Disabled ASAR packaging because node-llama-cpp's native binaries and the 2.4GB GGUF model cannot be loaded from inside a ASAR archive.
  • 7z distribution over NSIS — Windows NSIS installer has a 2GB hard mmap limit. At 3.5GB total, MIRA Mini exceeds it. The 7z archive bypasses this completely.
  • setAlwaysOnTop foreground hack — Windows suppresses focus-steal. We use a brief setAlwaysOnTop(true)/false cycle to physically force the window to the front when the deep link fires.
  • Native dispose chain — On engine shutdown, sequence → context → model are disposed in order to properly release VRAM and RAM back to the OS.

Reference

IPC Channel Map

Every communication between the renderer and main process goes through one of these channels. The preload script gates all access.

ChannelTypeDirectionDescription
renderer:readysendR → MRenderer signals boot complete. Main starts engine if onboarded.
model:statushandleR → MRenderer polls current engine status object.
model:statuseventM → RMain pushes status updates during engine lifecycle.
chat:sendhandleR → MSend a user message. Returns token count + elapsed ms on completion.
chat:streameventM → RStreams a single token chunk to the renderer during generation.
chat:stream-endeventM → RSignals generation is complete or was aborted.
chat:stophandleR → MAborts current generation via AbortController signal.
session:resethandleR → MClears and recreates the LlamaChatSession with identity primers.
chat:set-historyhandleR → MInjects a prior conversation into the session context before prompting.
system-prompt:sethandleR → MUpdates system prompt and resets session.
system-prompt:gethandleR → MReturns current active system prompt string.
history:listhandleR → MReturns sorted array of chat metadata from disk.
history:savehandleR → MWrites full chat object to userData/history/{id}.json.
history:loadhandleR → MReads and returns a single chat JSON by ID.
history:deletehandleR → MDeletes a single chat file from disk.
history:clearhandleR → MDeletes all history JSON files.
app:resethandleR → MFactory reset: clears history, removes .onboarded, relaunches app.
app:onboarded-successeventM → RSent after deep link fires, signals renderer to hide onboarding overlay.
log:entryeventM → RStreams each log entry to the Chronicle tab in real time.
log:get-pathhandleR → MReturns the path to the logs directory on disk.

Common Questions

FAQ

Answers to the most common questions from the team.

On first run, before onboarding is complete, the app window opens immediately in the foreground and shows a waiting overlay. The browser launches automatically with the onboarding page. This was a bug in earlier builds where the window was intentionally hidden — now it's always visible, using setAlwaysOnTop to forcefully claim focus when the deep link fires.
A 4-billion parameter quantized model requires approximately 2.4–3GB of RAM/VRAM to load. This is the floor for any local LLM of this quality. The engine is only initialized after onboarding completes, and all native memory (sequence, context, model) is explicitly disposed when the window closes via a proper dispose() chain.
Windows NSIS installers use a 32-bit memory-mapping API that has a hard 2GB file size limit. MIRA Mini's AI model alone is 2.4GB, making the total package ~3.5GB. We use the .7z format to bundle everything without splitting — users extract it once and run the exe directly.
Open the app, click Settings in the bottom-left, navigate to the Engine tab, and click Factory Reset App. This clears all dialogue history, removes the .onboarded flag, clears session storage, and relaunches the app fresh — triggering onboarding again.
Chats are stored as individual JSON files at %AppData%\mira-mini\history\{id}.json. Each file contains the full message array and metadata. Logs are at %AppData%\mira-mini\logs\mira-YYYY-MM-DD.log.
Yes. Edit config/default.json in the app's install directory and update modelPath to point to any compatible GGUF file in the models/ folder. The engine reads this config at initialization. Restart the app after changing it.
No, but strongly recommended. The app runs on CPU-only machines but generation will be slow (5–15 tokens/sec vs 30–60+ with a GPU). node-llama-cpp auto-detects CUDA (NVIDIA), Vulkan (AMD/Intel), or Metal (Apple Silicon) and uses hardware acceleration if available.
